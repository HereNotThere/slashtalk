# Info card — popover layout and data flow

The info card is the popover that opens when the cursor enters a chathead on the rail. It renders one of two layouts depending on `head.kind` (`agent` vs `user`), and pulls data from a mix of cached snapshots in main and live HTTP fetches against `/api/users/:login/*`.

This doc exists because the card has grown several conditional surfaces (header / Now / Today / PRs) and the data flow between main's caches and the renderer is non-obvious. If you're adding or changing a section, read this first.

## Card kinds

The renderer in [`src/renderer/info/App.tsx`](../src/renderer/info/App.tsx) branches on `head.kind`:

| Kind    | Layout (top → bottom)                                                       | Source file                                                            |
| ------- | --------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| `agent` | `AgentPanel` only — managed-agent status, no header                         | [`AgentPanel.tsx`](../src/renderer/info/AgentPanel.tsx)                |
| `user`  | `UserHeader` → `HierarchyDashboard` (Now / Today / PRs)                     | [`HierarchyDashboard.tsx`](../src/renderer/info/HierarchyDashboard.tsx) |

The `user` kind covers both the signed-in user (self) and peers; data sources are identical (`/api/users/:login/*`) and the renderer threads `isSelf` only into `UserHeader` for the self-mode location/time fallback.

## The hierarchy layout

`HierarchyDashboard` renders three sections separated by dividers:

```
┌─────────────────────────────────┐
│ UserHeader  (city, time, tokens)│
├─────────────────────────────────┤
│ NOW       (only if active)      │  ← live or recently-active session
├─────────────────────────────────┤
│ TODAY     (always rendered)     │  ← Claude-composed standup blurb
├─────────────────────────────────┤
│ PRs PUSHED (always rendered)    │  ← rows from /api/users/:login/prs
└─────────────────────────────────┘
```

### Now

Picks the most recent `BUSY`/`ACTIVE` session, or an `IDLE`/`RECENT` one whose `lastTs` is within the last 2h. `ENDED` sessions are excluded. Hidden entirely when nothing qualifies — see `pickNowSession()` in `HierarchyDashboard.tsx`.

Renders the analyzer's `description` (and only that — never the user's raw prompt as a fallback, which would be surveillance-y on peer cards), repo, total tokens, and a live "working now" shimmer when the session is `BUSY`/`ACTIVE`. While the analyzer hasn't produced a description yet, the section shows an italic "Summarizing…" placeholder so it never goes silent.

### Today (standup)

A 2-4 sentence Claude blurb generated by [`/api/users/:login/standup`](../../../apps/server/src/user/dashboard.ts). Always renders the section header so the card has consistent vertical rhythm; falls back to "Summarizing…" while loading and "Nothing shipped yet today." when the endpoint returns `summary: null`.

The standup prompt is biased toward shipped code — merged/closed PRs first, wrapped sessions second — and explicitly de-emphasizes live WIP since the **Now** section already covers that. The prompt is third-person past tense ("Shipped X, fixed Y") so the same blurb reads naturally whether the user is looking at their own card or a teammate's. See `STANDUP_SYSTEM` in [`apps/server/src/user/dashboard.ts`](../../../apps/server/src/user/dashboard.ts).

The output is rendered as inline markdown via the shared [`Markdown`](../src/renderer/shared/Markdown.tsx) component (`inline` mode), so two formatting conventions enforced by the prompt actually take effect in the UI:

- **PR references are markdown links.** Each PR row in the prompt carries `(url: …)`; the system rule tells Claude to emit `[#123](https://github.com/owner/repo/pull/123)` rather than a bare `#123`. The shared `Markdown` overrides `<a>` to route clicks through `window.chatheads.openExternal` (gated by `isSafeExternalUrl`) so links open in the user's browser, not the BrowserWindow.
- **Code-like tokens get backticked.** Filenames, paths, identifiers, shell commands, env vars, ports — `src/foo.ts`, `MyClass`, `bun run test`, `:3000`, `NODE_ENV`. PR numbers and repo names are explicitly excluded (PR numbers are links; repo names are prose).

### PRs pushed

PRs the target user authored that were updated inside the window, scoped to repos visible to the caller (caller's repos ∩ target's repos). Empty state ("No PRs in this window.") is informational rather than hidden so the section's intent stays visible.

## Time-window scope (the flag)

Both the standup and PRs sections accept a `scope` query param:

- `today` — start-of-day in the target user's IANA timezone (from `users.timezone`); falls back to UTC if unset.
- `past24h` — `now - 24h`.

Defaults to `today`. The flags live as constants in [`apps/desktop/src/main/windows/info.ts`](../src/main/windows/info.ts):

```ts
const DASHBOARD_PRS_SCOPE: DashboardScope = "today";
const DASHBOARD_STANDUP_SCOPE: DashboardScope = "today";
```

Each section's scope is independent so they can evolve at different cadences. Today this is a code-only flag — flip the constant and rebuild — not a UI toggle. The DST-aware boundary math lives in [`apps/server/src/util/time-window.ts`](../../../apps/server/src/util/time-window.ts) and is unit-tested.

## Data flow

```
overlay (renderer)
  └─ hover → IPC: heads:showInfo(headId, bubbleScreen)
       │
main (windows/info.ts)
  ├─ ensureInfoWindow / position / fade-in
  ├─ pushInfoShowSnapshot()  ──► info:show IPC
  │     payload: head, sessions, spotify, location, isSelf, dashboard
  ├─ fetchSessionsForHead(headId)         (cache: sessionCache, keyed by head.id)
  └─ fetchDashboardForLogin(login)        (cache: dashboardCache, keyed by login)
       └─ Promise.allSettled([
              backend.fetchUserPrs(login, scope),
              backend.fetchUserStandup(login, scope),
            ])
       │
info renderer (info/App.tsx)
  └─ onInfoShow → setHead/sessions/dashboard → React renders
       └─ useLayoutEffect → notifyInfoShowReady(measuredHeight)
            └─ main resizes the BrowserWindow before the first paint
```

### Caches

| Cache                                        | Where               | Invalidation                                                                                               |
| -------------------------------------------- | ------------------- | ---------------------------------------------------------------------------------------------------------- |
| `sessionCache: Map<headId, …>`               | `windows/info.ts`   | Per head on `session_updated` for that head; cleared on logout                                             |
| `dashboardCache: Map<login, …>`              | `windows/info.ts`   | Used as a **stale-while-revalidate** slot: every `showInfo` deletes the entry and refetches, so the renderer paints from the previous snapshot for instant feedback and re-renders once the fresh data lands. Also wholesale-cleared on `localRepos.onSelectionChange` and per-login on `session_updated` for the currently-shown head |
| `standupCache` (server)                      | `user/dashboard.ts` | TTL: 5 min per `(callerId, targetId, scope)`. Saves Anthropic budget on rapid re-hovers and peer re-views. **Bypassed and lazily deleted when `visibleRepoIds.length === 0`** (see "no claimed repos" below) so unclaiming all repos doesn't keep serving the pre-unclaim summary for up to 5 minutes |

Sessions and the dashboard are fetched in parallel with the initial snapshot push so the renderer paints loading placeholders, then re-renders once data lands. `info:show` ships with whatever's currently cached (often `null` on first hover) and a follow-up push lands when the parallel fetches settle.

#### Why the standup cache key includes `callerId`

The standup blurb is filtered by repos visible to the caller (caller's repos ∩ target's repos) so PR titles never leak from repos a viewer can't see. Cache keying on `(callerId, targetId, scope)` collapses to effectively `(targetId, scope)` for the typical case where everyone is on the same team (identical visible-repo overlap), while still preventing cross-team leaks. A teammate's daily blurb gets generated once per (caller, target) pair within a TTL window — not once per HTTP request.

### Empty states by section

| Section | `dashboard === null` (loading) | `dashboard.standup === null` / `prs === []`           |
| ------- | ------------------------------ | ----------------------------------------------------- |
| Now     | Hidden if no active session    | "Summarizing…" (italic) until the analyzer's `description` lands — never falls back to the raw user prompt |
| Today   | "Summarizing…"                 | "Nothing shipped yet today."                          |
| PRs     | "Loading…"                     | "No PRs in this window."                              |

The standup section deliberately renders an empty placeholder (rather than hiding) so the card's vertical layout stays stable and the user gets a clear "nothing happened" signal.

### "No claimed repos" CTA (self only)

Self-mode with zero rows in `user_repos` is a distinct state from a quiet day — "0 PRs and a null standup" is ambiguous (genuinely quiet vs. setup not done), so the server explicitly flags it and the renderer collapses every section into a single CTA.

Wire:

1. **Server.** Both `/api/users/:login/prs` and `/api/users/:login/standup` set `noClaimedRepos: true` on the response when `target.id === caller.id` and the resolved `visibleRepoIds` set is empty. The flag is never set on peer queries — peers with empty overlap already 403 in `resolveTarget`. The standup endpoint short-circuits **before** the `standupCache` lookup (and lazily deletes any stale entry at that key), so the moment a user unclaims all their repos, the next request returns the empty-state response rather than the previously-cached summary.
2. **Main.** `fetchDashboardForLogin` ORs the flag from both legs into `InfoDashboardData.noClaimedRepos` (OR rather than AND so a single failed leg doesn't silently drop the CTA).
3. **Renderer.** When `dashboard.noClaimedRepos === true`, `HierarchyDashboard` short-circuits to `<NoRepoSection>`: a single "No repos connected yet" panel with an **Open settings** button that calls `window.chatheads.openMain()` (the existing IPC that shows the main window).

Cache-invalidation interplay: the desktop's `dashboardCache` is a stale-while-revalidate slot — every show refetches in the background. It also wholesale-clears on `localRepos.onSelectionChange`, so toggling a repo in the rail tray (or removing a local repo entirely) immediately invalidates whatever was cached. Combined with the server-side pre-cache `visibleRepoIds` check, this means the empty-state CTA appears on the *next hover* after a repo change rather than waiting up to 5 minutes for either TTL.

## Access gate (peers)

[`/api/users/:login/prs`](../../../apps/server/src/user/dashboard.ts) and [`/api/users/:login/standup`](../../../apps/server/src/user/dashboard.ts) gate on the same `user_repos` overlap as `/api/users/:login/questions`:

- **Self** (`target.id === caller.id`): visible repo set = caller's `user_repos`.
- **Peer**: visible repo set = caller ∩ target. Empty overlap → `403 no_access`.

The visible-repo set is then used as the filter on PRs and sessions so a peer never sees mention of work on repos they don't share with the target. This is symmetric to [`core-beliefs #13`](../../../docs/design-docs/core-beliefs.md) (`user_repos` as the only authorization for cross-user reads).

## Known gaps

- **Private-repo PRs are invisible to the poller.** OAuth scope is `read:user read:org` (per [`core-beliefs #11`](../../../docs/design-docs/core-beliefs.md)). [`apps/server/src/social/pr-poller.ts`](../../../apps/server/src/social/pr-poller.ts) calls `GET /repos/:owner/:name/pulls` (403 without `repo` scope) and `GET /users/:login/events` (public events feed only — drafts on private repos don't appear). Result: the **PRs pushed** section stays empty for users whose only authored PRs are on private repos. Fix candidates: switch the user-events poll to `GET /users/:login/events/private`, or use the authenticated `GET /events` stream.
